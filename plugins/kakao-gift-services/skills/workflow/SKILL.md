# workflow

## 트리거

- 지라 URL/키: `GIFTPM-\d+`(개발), `QAGIFT-\d+`(QA)
- "선물하기" 키워드 + 작업 요청
- 현재 브랜치가 `(feat|hotfix|release)/(GIFTPM|QAGIFT)-...` 형태인 상태에서 "이어서", "마무리", "다음 커밋" 류 연속 작업 요청 (지라 키 생략되면 브랜치명에서 파싱)

## 브랜치 전략

지라 키가 확보되면 `jira_get_issue`로 이슈 내용을 확인한다.

- **타입**: 기본 `feat`. 핫픽스·긴급 배포 건이면 `hotfix`, 정기배포·릴리즈 건이면 `release`. 발화나 지라 내용으로 판단.
- **베이스**: 기본 `master`. 특정 프로젝트 브랜치에 종속된 작업이면 해당 브랜치.
- **main-lock 여부**: PR을 여러 번으로 쪼갤 것 같으면 `main-lock` 베이스를 먼저 만들고 그 아래 서브 브랜치에서 PR. 이름에 `lock`이 들어가면 GitHub 머지 후 자동삭제가 동작하지 않아 통합 브랜치로 재사용 가능. 에픽은 거의 확실히 `main-lock`, 그 외는 분할 여부로 판단.
- **네이밍**
  - 단일 PR: `<type>/<지라키>-<kebab-요약>` (예: `feat/GIFTPM-5811-chat-bridge`)
  - main-lock 베이스: `<type>/<지라키>-<kebab-요약>/main-lock`
  - main-lock 하위
    - 별도 지라 없음: `<type>/<지라키>-<kebab-요약>/<서브-kebab-요약>`
    - 관련 지라 있음: `<type>/<지라키>-<kebab-요약>/<관련지라키>-<서브-kebab-요약>`

## 브랜치 흐름

깃플로우 변형. 머지 경로: `feat → dev → beta → master`. 단, `feat`는 `dev`가 아닌 `master`에서 분기한다.

- `release/*`, `hotfix/*`는 `master`로 직접 PR.
- 리뷰를 받는 구간(같은 변경이 공용 브랜치에 처음 들어가는 지점):
  - `feat → dev`
  - `main-lock 하위 → main-lock`
- 이후 승격 PR(`feat → beta`, `feat → master` 등)은 동일 변경이 이미 리뷰를 통과했으므로 추가 리뷰 없이 진행.

## PR

- 베이스: 브랜치 흐름 참조. main-lock 하위는 `.../main-lock`. 특정 프로젝트 브랜치에 종속된 작업이면 해당 프로젝트 브랜치.
- 제목: `[<리뷰어수>] <제목>` 형식.
  - 기본 `[2]`, 간단한 변경이면 `[1]`.
  - 리뷰 불필요한 승격 PR(브랜치 흐름 참조)은 `[...]`를 생략하고 제목만.
- 본문 양식 없음. 단, **본문에 지라 링크 필수**.

```
## <지라키> 작업 완료

- PR: <URL>
- 브랜치: <브랜치명>
- 베이스: <베이스>
- 커밋: N개
  - <subject 1>
  - <subject 2>
- 변경 요약: <한두 줄>
- 검증: <결과 또는 미수행 사유>
```
