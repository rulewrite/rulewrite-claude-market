# workflow

## 트리거

- 지라 URL/키: `GIFTPM-\d+`(개발), `QAGIFT-\d+`(QA)
- "선물하기" 키워드 + 작업 요청
- 현재 브랜치가 `(feat|hotfix|release)/(GIFTPM|QAGIFT)-...` 형태인 상태에서 "이어서", "마무리", "다음 커밋" 류 연속 작업 요청 (지라 키 생략되면 브랜치명에서 파싱)

## 작동 방식: 에이전트 팀에 위임

이 워크플로우의 실무 — 지라 파악, 코드 탐색, 브랜치 생성, 구현, 검증, 커밋, PR — 는 에이전트 팀원이 한다. 메인은 팀장이다: 사용자와 대화하고, 팀원에게 일을 맡기고, 팀원이 보낸 결과를 사용자에게 전한다. 직접 지라를 읽거나 코드를 뒤지지 않고 되도록 idle하게 둔다.

위임은 raw 입력을 그대로 던지는 것이다. "좋은 프롬프트를 주려고" 메인이 먼저 지라·맥락을 읽고 분석하는 순간 그게 직접-수행이고, 컨텍스트가 차서 이후 판단이 흐려진다. 조사부터 팀원이 한다.

- **항상 `TeamCreate`로 팀을 만든다**: 작은 일은 팀원 한 명, 에픽·여러 갈래는 여러 팀원 — 규모만 다르다. 팀 없이 `Agent`만 쓰는 단발 서브에이전트는 쓰지 않는다.
- **디스패치 프롬프트 필수 요소**: ① 이 스킬 파일의 절대경로를 주고 "Read로 먼저 읽고 규칙대로 진행"을 지시 — 새 팀원은 이 워크플로우를 모르므로, 빠지면 generic 브랜치·PR이 나온다. ② 지라 키와 사용자가 준 원문 맥락. ③ 워크트리 격리. ④ 모든 보고는 `SendMessage`로 — 팀원이 일반 텍스트로 답하면 메인에 전달되지 않는다.
- **워크트리 격리**: `isolation: "worktree"`로 띄운다. 팀원이 별도 워크트리·자기 브랜치에서 작업하고 path·branch를 반환하면, 메인은 그 브랜치로 PR을 잇는다. 격리 없이 병렬로 띄우면 한 팀원의 미커밋 변경이 다른 팀원의 `checkout -b`에 끌려가 git 상태가 망가진다.
- **사용자 합의**: 팀원은 작업 중에도 `SendMessage`로 조정되니, 합의가 필요한 결정은 메인이 사용자와 정리해 팀원에 전달한다.

## 브랜치 전략

아래는 팀원이 따를 브랜치 규칙이다. 팀원이 지라를 읽고 이 규칙으로 전략을 정해 진행하며, 사용자 합의가 필요한 결정은 메인을 거친다.

- **타입**: 기본 `feat`. 핫픽스·긴급 배포 건이면 `hotfix`, 정기배포·릴리즈 건이면 `release`. 발화나 지라 내용으로 판단.
- **베이스**: 기본 `master`. 특정 프로젝트 브랜치에 종속된 작업이면 해당 브랜치.
- **main-lock 여부**: PR을 여러 번으로 쪼갤 것 같으면 `main-lock` 베이스를 먼저 만들고 그 아래 서브 브랜치에서 PR. 이름에 `lock`이 들어가면 GitHub 머지 후 자동삭제가 동작하지 않아 통합 브랜치로 재사용 가능. 에픽은 거의 확실히 `main-lock`, 그 외는 분할 여부로 판단.
- **네이밍**
  - 단일 PR: `<type>/<지라키>-<kebab-요약>` (예: `feat/GIFTPM-5811-chat-bridge`)
  - main-lock 베이스: `<type>/<지라키>-<kebab-요약>/main-lock`
  - main-lock 하위
    - 별도 지라 없음: `<type>/<지라키>-<kebab-요약>/<서브-kebab-요약>`
    - 관련 지라 있음: `<type>/<지라키>-<kebab-요약>/<관련지라키>-<서브-kebab-요약>`

## 브랜치 흐름

깃플로우 변형. 머지 경로: `feat → develop → beta → master`. 단, `feat`는 `develop`이 아닌 `master`에서 분기한다.

- `release/*`, `hotfix/*`는 `master`로 직접 PR.
- 리뷰를 받는 구간(같은 변경이 공용 브랜치에 처음 들어가는 지점):
  - `feat → develop`
  - `main-lock 하위 → main-lock`
- 이후 승격 PR(`feat → beta`, `feat → master` 등)은 동일 변경이 이미 리뷰를 통과했으므로 추가 리뷰 없이 진행.

## PR

- 베이스: 브랜치 흐름 참조. main-lock 하위는 `.../main-lock`. 특정 프로젝트 브랜치에 종속된 작업이면 해당 프로젝트 브랜치.
- 제목: `[<리뷰어수>] <제목>` 형식.
  - 기본 `[2]`, 간단한 변경이면 `[1]`.
  - 리뷰 불필요한 승격 PR(브랜치 흐름 참조)은 `[...]`를 생략하고 제목만.
- 본문 양식 없음. 단, **본문에 지라 링크 필수**.
