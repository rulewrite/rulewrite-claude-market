# planning

구현이 아닌 **계획 단계**에서 사용.

## 전체 흐름

A → B → C → D → E, 5단계. **각 단계 끝에 사용자 합의(체크포인트)** 후 다음으로. 중간 수정 요청은 해당 단계만 재실행. 체크포인트는 생략 금지.

| 단계 | 출력 |
|---|---|
| A 기획서 읽기 | 슬라이드별 변경점 요약 + 질문 목록 |
| B 코드 탐색 | 재사용 / 리팩 / 신규 매핑 |
| C 태스크 분해 + 지라 생성 | 서브 이슈 N개 |
| D 버퍼 이슈 | 버퍼 이슈 1개 |
| E 일정 산출 | 역산 착수일 또는 최단 완료일 |

---

## A. 기획서 읽기

**입력**: Google Slides / Docs URL.
- `docs.google.com/presentation/d/<ID>/...` 또는 `/document/d/<ID>/...` 에서 `<ID>` 추출
- kko.to 같은 단축 URL은 해석 실패 가능성 있음 → 원 URL 요청
- `mcp__claude_ai_Google_Drive__read_file_content` 로 읽기. 스코프 부족 시 재인증 안내

**요약 포맷**:
- 배경/현황 지표 → 개선 목표
- 변경 사항: 슬라이드 번호와 함께 AS-IS / TO-BE 구분
- 관련 BE·디자인·다른 지라 링크 수집
- 모호한 부분은 별도 "질문" 목록으로 뽑아 체크포인트에서 확인

**체크포인트**: 요약에 누락/오해 있는지, 질문 답변 받고 확정.

---

## B. 코드 탐색

**목적**: 기획 요구가 실제 코드 어디에 걸리는지, 신규/재사용/리팩 경계 판단.

**방법**:
- Explore 에이전트로 병렬 탐색. 키워드는 기획서에서 추출 (예: 배너 이름, 컴포넌트 성격, API 명)
- 결과를 **재사용 가능 / 리팩 필요 / 신규 구현** 3범주로 분류
- 상위 레벨 파일 경로 + 핵심 함수/컴포넌트 이름 + 짧은 코멘트 형태로 정리

**체크포인트**: 사용자가 외부 컨텍스트 추가할 수 있는 지점 — 예: "이 컴포넌트는 다른 팀이 공통화 중이니 건드리지 마", "이 API는 곧 deprecate".

## C. 태스크 분해 + 지라 서브태스크 생성

### 1) 분해 원칙
- 독립적으로 PR 낼 수 있는 단위
- **같은 파일/컴포넌트를 여러 태스크가 수정한다면 묶어 1개로** (리뷰·테스트 오버헤드 절감)
- 선후 의존성 명시 (앞 태스크에서 공용화한 결과를 뒤 태스크가 쓰는 경우)

### 2) 공수 산정
- **0.5d 단위** (4h 단위). 1d = 8h
- 공용화로 뒤 태스크가 줄면 앞 태스크에 포함하고 뒤는 깎아 반영

### 3) 지라 이슈 양식

**Summary**:
```
p<슬라이드번호>[, <슬라이드번호>...] [영역] 설명
```
- `p` 는 기획서 페이지/슬라이드 번호. 여러 개면 콤마
- `[영역]` = 페이지 단위로 유연. 예: `[홈]`, `[선물함]`, `[어드민]`, `[홈, 선물함]`. 애매하면 생략

예시:
```
p10, 15 VIP 페이지 프로덕트 컴포넌트 추출하여 바텀시트에서 재사용
p8, 9, 10, 19 [홈, 선물함] 쿠폰 다운로드 성공 바텀시트 연결, 실패 팝업
p17 [선물함] 잠재 VIP 문구 변경
```

**Issue Type**: `Story`

**Estimated**: `originalEstimate` 와 `remainingEstimate` 같은 값 (0.5d 단위)

**상위 연결**:
- 상위가 **Epic** → `epic_link`
- 상위가 **Task/Story** → `parent`
- 상위 이슈 타입은 `mcp__jira__jira_get_issue` 로 먼저 확인

**Description**:
- `h2` / `h3` 섹션. "포함 작업", "비고" 정도 구조
- 비고에는 의존성(BE 선행, 공용화 대기, 다른 PR 연계) 명시
- 참고 링크(관련 지라, 위키, 기획서 슬라이드) 첨부

### 4) 생성 순서
1. 분해/공수 테이블을 사용자에게 먼저 제시
2. 합의 후에만 `jira_create_issue` 병렬 호출
3. 생성 완료 후 합계(개발 공수) 보고

### 5) 이슈 수정/이관/삭제
- 이관(A를 B에 흡수)은 A의 내용이 B에 반영된 것 확인 후 A 삭제
- 이미 생성된 이슈의 포맷 소급 적용 여부는 **사용자에게 물어보기**

## D. 버퍼 산정 + 버퍼 이슈

**규칙 출처**: FE팀 위키 — https://wiki.daumkakao.com/spaces/feplatform/pages/1384900942

**공식**:
- 기본 버퍼 = 개발 공수 × **25%**
- 회의/코드리뷰 = ⌈개발 공수 ÷ 3.5d/주⌉ × **1.5d/주** (1d 회의 + 0.5d 코드리뷰)
- 주번 = 주번 일수 × **50%** — 사용자에게 주번 일수 확인. 없으면 0
- **총 버퍼 = 세 항 합, 0.5d 단위로 반올림**

**버퍼 이슈**:
- Summary: `[버퍼] 프로젝트 버퍼 (회의/코드리뷰 + 25% 불확실성)` (고정)
- Issue Type: `Story`
- Estimated: 반올림된 총 버퍼
- Description: **계산식 금지**. 사용 방법만 한두 줄 (예: "프로젝트 버퍼. 실제 소진 시마다 remaining 차감")
- **에픽(또는 상위 이슈)마다 1개**
- 상위 연결: 개발 이슈와 동일

**체크포인트**: 계산 결과 보고 → 합의 → 이슈 생성.

---

## E. 일정 산출

**총 공수** = 개발 서브태스크 Estimated 합 + 버퍼 이슈 Estimated

**캘린더 규칙**:
- 주 5일, **한국 공휴일 제외**
- 대체공휴일 포함 (일요일에 겹친 명절 등)
- 음력 기반 공휴일(설날·부처님오신날·추석)은 양력 환산이 연 단위로 다르니 실행 시점 연도 기준으로 확인

**두 방향**:
- **목표 완료일 주어짐** → 완료일부터 워킹데이로 공수만큼 역산 → 착수일 산출
- **목표 완료일 미정** → 마크업/디자인 완료일 다음 영업일 기준 착수 가정 → 완료일 산출

**출력**: 주별 표 — 근무일 / 휴일 / 가용 일수 / 누적.

**체크포인트**: 일정 제시 후 개인 휴가·외부 의존 일정 반영 기회 제공.

## 주요 도구

- Google Drive: `mcp__claude_ai_Google_Drive__read_file_content` (fileId 기반)
- 지라 조회: `mcp__jira__jira_get_issue`, `mcp__jira__jira_search`
- 지라 생성/수정/삭제: `mcp__jira__jira_create_issue`, `mcp__jira__jira_update_issue`, `mcp__jira__jira_delete_issue`
- 코드 탐색: `Agent(subagent_type=Explore)`

## 선물하기 FE 전제

- 지라 프로젝트 키: `GIFTPM` (개발), `QAGIFT` (QA)
- 위키 도메인: `wiki.daumkakao.com`
- 1d = 8h, 공수는 0.5d 단위

## 용어

선물하기 내부 축약어·고유명사가 나오면 추측 금지. 플러그인 루트의 `references/glossary.md` 조회 → 없으면 사용자에게 확인 → 확정 의미는 glossary에 추가 제안.

## 금지 사항

- 기획서 전문 요약 없이 복사
- 지라 summary에 자체 번호(#1, #2…) 붙이기
- 버퍼 이슈 description에 계산식 쓰기
- 사용자 합의 없이 지라 이슈 생성/수정/삭제
- 확인 가능한 사실을 "모르겠다"로 회피
