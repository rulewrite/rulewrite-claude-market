# weekly-worklog

업무 단위(PR 또는 지라 이슈) 로그를 노션에 쌓는 스킬. 문서 구조는 사용자마다 달라 고정 포맷을 강제하지 않고 기존 엔트리를 읽어 그대로 맞춘다.

## 기본 흐름

1. `~/.claude/state/weekly-worklog/target.json` 로드 → 없으면 "첫 실행" 절차.
2. 기록할 작업 정보 확보.
3. 기존 문서 구조를 보고 이번 엔트리가 들어갈 위치를 판단한다. 해당 위치(주차 섹션, 월 단위 로우 등 기존 문서가 쓰는 묶음 단위)가 아직 없으면 기존 포맷 그대로 새로 만든다.
4. 학습한 엔트리 포맷대로 항목 append.
5. 대화창에 짧게 보고.

## 입력

기본은 작업 1건 = 로그 1건. PR URL/번호 또는 지라 URL/키를 받으면 그걸로 기록한다. URL 없이 "방금 한 X 작업 써둬"처럼 자유 입력으로 설명해줘도 된다.

PR인 경우 `gh pr view <url> --json number,title,url,body,mergedAt,baseRefName,headRefName,commits`로 정보 확보. 지라인 경우 Jira MCP의 `jira_get_issue`로 확보.

확보해두는 정보:

- 제목, URL/키, 완료 시점(PR `mergedAt`, 지라 resolution date)
- PR ↔ 지라 상호 링크(브랜치명·본문에 지라 키가 있거나, 지라 이슈에 PR이 달려 있으면 함께 기재)

어떤 필드를 어떻게 쓸지는 학습한 포맷에 맞춘다.

## 대상 문서 관리

대상 노션 문서는 첫 실행 때 한 번 확정해 저장한다. 매 실행 검색하면 토큰 낭비이고 오탐 위험이 있다.

저장 경로: `~/.claude/state/weekly-worklog/target.json`

```json
{
  "kind": "page | database",
  "id": "<notion id>",
  "url": "<notion url>",
  "title": "<문서 제목>",
  "format_hint": "<포맷 요약 메모 — 선택>"
}
```

### 첫 실행

`target.json`이 없거나 사용자가 "워크로그 문서 다시 정해줘" 류로 재설정을 요청한 경우.

1. 사용자에게 문서 URL 또는 제목·키워드를 묻는다.
2. URL을 주면 그대로 확정. 키워드면 노션에서 검색해 후보를 나열한다.
3. 후보가 여럿이면 사용자가 고르도록 한다.
4. 확정 후 위 경로에 저장.

### 포맷 학습

확정 후 최근 엔트리 1~2개를 실제로 읽어 다음을 파악한다.

- **묶음 단위**: 주차 섹션인지, 월 단위 DB 로우인지, 개별 페이지인지
- **묶음 표기**: `2026-W16`, `4월 3주차`, 날짜 범위 등 어떤 방식인지
- **항목 구성**: PR 링크, 지라 키, 한 줄 요약, 상세, 결과/지표 중 무엇을 어떤 순서로 쓰는지
- **언어체**: 경어/평어, 완결문/명사형 등

학습 요약은 `format_hint`에 남겨 다음 실행 비용을 줄인다. 다만 append 직전 마지막 엔트리 하나는 매번 확인해 포맷 변경 여부를 짚는다.

## 애매한 경우

- 기록 대상이 불분명하거나 소스가 비면 자동으로 쓰지 말고 사용자에게 확인한다.
- 동일 PR이 이미 기록돼 있으면 중복 추가하지 말고 사용자에게 알린다(수정/스킵/강제 추가 중 선택).

## 결과 보고

대화창에만 짧게 보고. 노션 댓글 금지.

```
## 워크로그 기록 완료

- 문서: <제목> (<URL>)
- 위치: <엔트리가 들어간 섹션/로우 식별자>
- 추가 항목: <한 줄 요약>
```

## 운영 메모

- **노션 접근 수단 필요**: 노션 문서를 읽고 쓸 수 있는 수단(MCP, CLI, API 등 무엇이든)이 현재 환경에 준비돼 있어야 한다. 없으면 먼저 사용자에게 설치·인증을 안내한다.
- **개인 설정은 레포에 남지 않는다**: `target.json`은 `~/.claude/state/` 아래에 저장되므로 마켓플레이스 레포에 커밋되지 않는다.
